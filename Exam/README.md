# Промежуточная аттестация (экзамен)

**Статус:** регламент промежуточной аттестации дисциплины.  
**Назначение:** проверить способность связать результаты нескольких модулей в целостное инженерное решение в роли Data Engineer.

Регламент и критерии оценивания — зона ответственности эксперта по промежуточной аттестации; тексты кейсов банка — в согласовании с разработчиком экзаменационных материалов ([team](../team/README.md)).

## 1. Формат

| Параметр | Решение |
|---|---|
| Форма | **устный** экзамен |
| Материалы и устройства | **не допускаются** (закрытый устный ответ: без конспектов, ноутбука, телефона, ИИ-ассистентов и иных подсказок) |
| Набор | **5 вопросов** из [банка интеграционного контроля](../resources/problem-banks/integrated-control-questions.md) |
| Готовые билеты в репозитории | не формируются: комиссия составляет набор из банка по правилам ниже |
| Длительность | определяется набором из 5 устных ответов и уточняющими вопросами комиссии; ориентир — **25–40 минут** на студента |

## 2. Правила составления набора из 5 вопросов

1. Берите вопросы из банка **И-01—И-16**; альтернативы **У-46—У-57** допустимы как замена близкого по смыслу основного, но **не как смысловой дубль** другого вопроса того же набора.
2. В одном наборе **не повторяйте** один код и не ставьте два вопроса, которые по сути проверяют одно и то же решение.
3. **Рекомендуется** покрыть разные результаты обучения или разные аспекты контура данных (источники / хранение / интеграция и конвейер / качество и безопасность / ИИ-потребление или облако).
4. Эталон для комиссии — колонка «Ожидаемый ответ» и «Методическое обоснование» в банке; отдельный закрытый ключ не вводится.

## 3. Шкала за один вопрос: 0 / 0,5 / 1

За каждый вопрос выставляется **один** из трёх баллов. Оценка отражает **качество устного инженерного ответа по существу кейса**, а не число произнесённых терминов и не название конкретного вендора.

**Рекомендуется** называть классы решений и при необходимости пример стека; **обязательным** является обоснованная инженерия, а не бренд продукта.

### 1 — верный ответ

Ответ засчитывается как верный, если **одновременно** выполнено:

1. Ответ **по существу** поставленного кейса (студент решает заданную ситуацию, а не пересказывает соседнюю тему).
2. Явно или по ходу ответа зафиксированы **релевантные допущения** (что известно / неизвестно, кого считает потребителем, какие ограничения принимает).
3. Предложено **связное решение**: видны связи между элементами контура (источник → получение/интеграция → качество → хранение/витрина → потребитель / надёжность / доступ), а не список несвязанных названий.
4. Названы **существенные риски или ограничения** и **способ проверки** результата (метрика, сверка, тест, критерий приёмки, наблюдение) — в объёме, которого требует вопрос.
5. Нет **критической** инженерной ошибки (такой, при которой предложенное решение заведомо непригодно или опасно для заявленной цели).

Уточняющий вопрос комиссии может помочь студенту договорить недостающий элемент; если после уточнения критерии выше выполнены — ставится **1**.

### 0,5 — частичный ответ

Ставится, если направление мысли **в целом верное**, но ответ **неполон или неустойчив**, например:

- раскрыта только **часть** ожидаемых элементов из банка (есть «что», почти нет «почему / как проверить / какой риск»);
- перечислены компоненты или технологии **без связей** и без критериев выбора;
- упущено ограничение или риск, **без которого** решение для данного кейса существенно неполно (доступ, свежесть, идемпотентность, версия документа, ПДн и т.п. — по смыслу вопроса);
- формулировки общие («обеспечим качество», «возьмём облако», «настроим RAG») **без** критериев, правил или порядка действий;
- есть заметные неточности, которые **не разрушают** общую логику, но снижают доверие к решению;
- после уточняющего вопроса студент улучшает ответ лишь частично.

Частичный балл — не «утешительный»: он означает, что студент **понимает задачу**, но **не довёл** ответ до проверяемого инженерного уровня.

### 0 — неверный ответ

Ставится, если имеет место **хотя бы одно** из следующего:

- ответ **не по теме**, отказ по сути или подмена вопроса другой темой;
- **грубая подмена понятий**, из‑за которой решение бессмысленно (например, отождествление несовместимых классов хранилищ или ролей без различения там, где вопрос это требует);
- предложенное решение **противоречит** условиям кейса и студент не исправляет это при уточнении;
- только **перечень продуктов/аббревиатур** без инженерной логики применительно к кейсу;
- **критическая** ошибка (безопасность, уничтожение воспроизводимости, заведомо неверный поток данных и т.п.), которую студент не осознаёт и не корректирует;
- ответ сводится к заученному шаблону **без привязки** к данным и ограничениям данного кейса.

Если студент молчит по вопросу или отвечает односложно без содержания — **0**.

### Как пользоваться шкалой на устном экзамене

1. Сверить ответ с колонкой «Ожидаемый ответ» в банке — как с **ориентиром полноты**, а не с дословным текстом.
2. Задать **1–2 уточнения**, если ответ близкий к границе 0,5 / 1 или 0 / 0,5.
3. Выставить **один** балл 0 / 0,5 / 1; не дробить иначе и не суммировать «подкритерии» внутри вопроса.
4. Не повышать балл за упоминание вендора или ИИ; не снижать только за то, что стек назван обобщённо (класс решения), если логика верна.

## 4. Итог экзамена и связь с БРС

| Показатель | Значение |
|---|---|
| Сырая сумма за экзамен | сумма баллов по 5 вопросам, диапазон **0–5** |
| Минимум для засчитывания экзамена | **не менее 3,0 из 5** |
| Перевод в баллы дисциплины (БРС) | **баллы_БРС = сырая_сумма × 4** (максимум **20**) |
| Минимум в шкале БРС | **12 из 20** (соответствует 3 из 5) |

Примеры: 3,0 → 12; 3,5 → 14; 4,0 → 16; 5,0 → 20.  
Если сырая сумма **меньше 3,0**, экзамен **не засчитывается** для положительной отметки по дисциплине (даже при высокой сумме за семестр по другим видам контроля) — см. [БРС](../docs/points-rating-system.md).

Пояснение «порога»: порог — это **минимум по экзамену** (3 из 5), без которого итоговая отметка по дисциплине не может быть положительной; отдельно действует общий минимум **60** баллов за дисциплину и минимум по лабораторным.

## 5. Требования к ответу студента

Студент устно:

- фиксирует допущения;
- предлагает решение по кейсу;
- называет компромиссы и риски;
- указывает способ проверки результата;
- поясняет роль Data Engineer там, где это следует из вопроса.

Конкретный поставщик облака/СУБД **рекомендуется** как иллюстрация; достаточным является корректный выбор **класса** решения с обоснованием.
