## 1. Цель работы

Целью лабораторной работы является формирование у студента навыков структурированного, экспертно обоснованного выбора ETL/ELT-системы для сбора и выверки данных из разнородных автоматизированных информационных систем (АИС) предприятия с использованием метода покритериального (многокритериального) сравнения.

По итогам выполнения работы студент должен уметь:

1. формулировать критерии сравнения программных систем по группам (функциональные, технические, ценовые, критерии информационной безопасности)
2. определять экспертные веса критериев с учётом их относительной значимости для конкретной задачи
3. строить и заполнять матрицу покритериального сравнения нескольких альтернатив
4. рассчитывать интегральную (взвешенную) оценку каждой альтернативы и делать обоснованный выбор
5. аргументированно связывать выбор ETL-системы с целями последующей аналитики и прогнозирования

## 3. Порядок выполнения работы

Задание: сравнить ETL-системы для сбора и выверки данных из перечисленных выше АИС и обосновать выбор одной из них методом покритериального сравнения. Работа выполняется в два этапа, каждый из которых завершается обсуждением результата с преподавателем перед переходом к следующему шагу.

### Этап 1. Формирование критериев сравнения и определение их весов

Прежде чем сравнивать конкретные ETL-системы, необходимо определить, по каким параметрам и с какой значимостью они будут оцениваться. Это ключевой шаг: неправильно подобранные или неверно взвешенные критерии могут привести к выбору технически неподходящего решения, даже если формально расчёт «сойдётся».

#### 3.1.1. Выберите минимум 3 альтернативы ETL/ELT-систем

Изучите рынок ETL/ELT-инструментов и выберите не менее трёх систем-кандидатов, потенциально подходящих для интеграции указанных АИС (Oracle ERP, Oracle CRM, PostgreSQL-система прогноза). Примеры систем для рассмотрения (допускается заменить на другие по согласованию с преподавателем):

1. Apache NiFi — open-source, потоковая и пакетная интеграция, широкий набор коннекторов
2. Apache Airflow (в связке с кастомными операторами) — оркестрация ETL-пайплайнов на Python
3. Talend Open Studio / Talend Data Fabric — визуальное проектирование ETL, готовые коннекторы к Oracle/PostgreSQL
4. Informatica PowerCenter — промышленное решение с широкой поддержкой корпоративных СУБД
5. Fivetran / Airbyte — managed/open-source ELT-платформы с готовыми коннекторами

#### 3.1.2. Сформируйте критерии по четырём группам

Разбейте критерии сравнения на четыре обязательные группы, отражающие разные аспекты пригодности системы. Перечень групп критериев можно расширять. Ниже приведён минимальный рекомендуемый набор — вы должны дополнить его минимум 40 собственными критериями в каждой группе, обосновав их значимость именно для данного кейса (интеграция Oracle ERP/CRM + PostgreSQL-системы прогноза).

*Рисунок (к построению студентом): иерархия критериев покритериального сравнения ETL-систем (группы → критерии → веса).*
*Рисунок 2 — Иерархическая структура критериев покритериального сравнения ETL-систем*

**Группа «Функциональные»:**

1. поддержка коннекторов к Oracle «из коробки»
2. поддержка gRPC протоколов
3. наличие встроенных механизмов data quality (дедупликация, matching, устранение дублей ФИО/контрагентов)
4. поддержка как пакетной (batch), так и потоковой (streaming) обработки

**Группа «Технические»:**

1. производительность и пропускная способность при больших объёмах данных (задать конкретные цифровые диапазоны параметров)
2. горизонтальная масштабируемость
3. простота развёртывания и сопровождения (DevOps-нагрузка)
4. совместимость с существующей инфраструктурой предприятия

**Группа «Ценовые»:**

1. стоимость лицензии/подписки (задать диапазон цен)
2. совокупная стоимость владения (TCO) с учётом затрат на инфраструктуру и поддержку (задать диапазон цен)
3. модель лицензирования (per-node, per-connector, open-source)

**Группа «Информационной безопасности (ИБ)»:**

1. шифрование данных при передаче и хранении
2. управление доступом и аудит действий пользователей
3. соответствие требованиям 152-ФЗ при обработке персональных данных клиентов (CRM)

Каждый критерий должен быть измеримым, чтобы для определения соответствия ему можно было однозначно ответить «да» или «нет».

Для экономии усилий при сравнении решения допускается выделять группу стоп-критериев (например, доступность решения для нашей страны и т.п.).

#### 3.1.3. Определите экспертные веса критериев

Веса отражают относительную значимость каждого критерия внутри группы для конкретной бизнес-задачи (сбор данных для последующего прогноза сбыта и закупок). Рекомендуемый порядок действий:

1. для каждого критерия определите его вес (от 0 до 1 с шагом 0,1). Составьте таблицу сравнения
2. зафиксируйте и обоснуйте выбранные веса текстом — почему именно такое распределение значимости выбрано для данного кейса

Пример применения рубрики к конкретному критерию: для критерия «поддержка коннекторов к Oracle и PostgreSQL» у Apache NiFi есть встроенные процессоры QueryDatabaseTable и ExecuteSQL с готовыми JDBC-коннекторами к обеим СУБД — это соответствует баллу 1. Если бы для подключения к Oracle требовалась установка стороннего JDBC-драйвера вручную и написание кастомного SQL-скрипта, это соответствовало бы баллу 0,5. Источниками для выставления оценок служат официальная документация системы, независимые технические обзоры и, при наличии доступа, тестовое развёртывание, экспертная оценка.

| Балл | Критерий выполнения |
|---|---|
| 1 | Функция/характеристика поддерживается «из коробки», без дополнительной настройки или кода |
| 0,8 | Функция поддерживается через штатный плагин/коннектор, требует конфигурации, но не написания кода |
| 0,6 | Функция реализуется через дополнительный код или скрипт (custom-разработка) |
| 0,4 | Функция технически возможна через обходное решение (workaround), но не рекомендуется производителем |
| 0 | Функция не поддерживается и не может быть реализована в рамках системы |

> **Результат этапа 1:** перечень критериев по группам, таблица весов критериев внутри групп, текстовое обоснование распределения весов (0,5–1 страница). Обсудите набор критериев и веса с преподавателем до перехода к этапу 2.

### Этап 2. Построение матрицы покритериального сравнения систем

На этом этапе вы переходите от определения критериев к количественной оценке каждой из выбранных ETL-систем и получению итогового обоснованного выбора.

#### 3.2.1. Оцените каждую систему по каждому критерию

Для каждой альтерна��ивы (минимум 3 системы, выбранные на этапе 1) выставьте оценку его соответствия каждому критерию («да» = 1, «нет» = 0, «нет данных» = 0).

Оценки должны опираться на реальные технические характеристики систем (документация, открытые сравнения, тестовые развёртывания), а не быть произвольными — приложите к отчёту таблицу по каждому решению с кратким обоснованием оценки каждого критерия.

#### 3.2.2. Рассчитайте взвешенную (интегральную) оценку

По каждому сравниваемому решению просуммируйте оценки его соответствия критериям с учётом их веса («да» = 1, «нет» = 0, «нет данных» = 0). Решение, получившее наибольшую суммарную оценку, является наиболее подходящим.

Итоговая оценка альтернативы рассчитывается по формуле:

\( S = \sum_{i} (\text{вес критерия}_i \times \text{оценка альтернативы по критерию}_i) \) (1)

где сумма берётся по всем критериям всех четырёх групп. Система с максимальным значением \( S \) [1] признаётся предпочтительной по результатам покритериального сравнения.

Оформите сравнение в виде таблицы со следующими полями:

| ID критерия | Критерий | Вес | Система А (NiFi) | Система Б (Talend) |
|---|---|---|---|---|
| | | | | |

*ID критерия — уникальный номер для удобства обработки (не нужно располагать критерии по порядку возрастания их ID)*

> **Результат этапа 2:** полная матрица сравнения по всем критериям из этапа 1, расчёт интегральной оценки для каждой системы, итоговый вывод о предпочтительной альтернативе с текстовым обоснованием (0,5–1 страница). Обсудите итоговый выбор с преподавателем.

## 4. Связь с ИИ-компонентой курса

С использованием метода покритериального сравнения обосновывается выбор ETL-системы для построения системы сбора данных, обеспечивающей повышение качества систем прогноза сбыта и прогноза закупок материалов и комплектующих. Качественно интегрированные и выверенные данные из ERP, CRM и системы прогноза напрямую влияют на точность моделей прогнозирования, используемых в аналитическом контуре предприятия.

## 5. Требования к отчёту

1. титульный лист (ФИО студента, группа, номер и тем�� лабораторной работы)
2. перечень выбранных для сравнения ETL-систем (не менее трёх) с кратким описанием каждой
3. критерии сравнения по четырём группам с обоснованием и таблицей весов (результат этапа 1)
4. полная матрица покритериального сравнения с расчётом интегральной оценки (результат этапа 2)
5. итоговый обоснованный выбор ETL-системы с указанием её преимуществ для задачи сбора данных под прогноз сбыта и закупок
6. краткие выводы по работе (0,5 страницы)

> Отчёт оформляется в электронном виде (DOCX или PDF), объём — 6–10 страниц, шрифт Times New Roman 14 пт.
