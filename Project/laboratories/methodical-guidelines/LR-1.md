## 1. Цель работы

Целью лабораторной работы является формирование у студента практических навыков проектирования гибридного хранилища данных (векторная база и структурированная витрина данных) для использования в RAG-агенте (Retrieval-Augmented Generation), а также освоение методов архитектурного описания информационных систем с использованием нотации C4 и ER-диаграмм.

По итогам выполнения работы студент должен уметь:

1. формулировать техническое задание на проектирование ИИ RAG-агента по ГОСТ 34 серии
2. определять состав данных, подлежащих векторизации, и данных, остающихся в структурированном виде
3. строить архитектурные диаграммы уровня Context и Container в нотации C4
4. разрабатывать ER-диаграмму реляционной витрины данных
5. обосновывать выбор технологического стека (векторная БД, ETL-инструменты, СУБД)

## 2. Бизнес-контекст и входные данные

Заказчик — агентство по продаже недвижимости, планирующее внедрить RAG-агента для консультирования потенциальных клиентов. Агент должен прогнозировать цену объекта недвижимости в зависимости от рынка и локации, оценивать потенциал изменения цены, а также вести предварительное обсуждение и обоснование предложения с клиентом.

### 2.1. Входные данные

1. база рекламных буклетов строительных компаний в разных форматах (PDF, DOCX, PPTX, TIFF)
2. API Яндекс.Карт — для получения данных о близости объектов к инфраструктуре
3. API агрегатора объявлений по аренде и покупке недвижимости или альтернативный датасет выгрузки объявлений о продаже квартир по районам города
4. данные по динамике изменения цены по районам и перспективам изменения района (определяются студентом при разработке ТЗ)
5. данные по отзывам и обсуждениям в соцмедиа (определяются студентом при разработке ТЗ)

### 2.2. Факторы, которые должно учитывать хранилище

1. предложения на рынке — цена по типам недвижимости (первичное/вторичное жильё)
2. близость к станциям метро
3. инфраструктура района — магазины, школы, поликлиники, торгово-развлекательные центры, парки и т.п.
4. данные по отзывам и обсуждениям в соцмедиа об объекте и районе
5. динамика изменения цены по районам
6. перспективы изменения района (по данным генплана развития территории)

> **Примечание.** Источник данных об изменении цены по районам (например, API irin.ru) может быть недоступен. Если подключение невозможно, используйте резервный источник — открытые агрегированные отчёты аналитических порталов (например, Домклик, Яндекс.Недвижимость) или смоделированный тестовый датасет, оформив это как явное допущение в ТЗ.

## 3. Порядок выполнения работы

Работа выполняется поэтапно и построена как мини-проект: три этапа, каждый завершается кратким обсуждением с преподавателем перед переходом к следующему шагу.

### Этап 1. Составление технического задания (ТЗ)

#### 3.1.1. Пользовательские сценарии и диаграмма C4 уровня Context

Опишите 2–3 конкретных диалога, которые RAG-агент должен уметь поддерживать. Например:

1. «Клиент интересуется трёхкомнатной квартирой в определённом районе — агент должен подобрать варианты с учётом бюджета, близости к метро, инфраструктуры, потенциала её развития и отзывов в соцмедиа по объекту и району»
2. «Клиент спрашивает, вырастет ли цена на квартиру в следующем году — агент должен опираться на историю цен по району и данные генплана»
3. «Клиент сравнивает два района по уровню инфраструктуры — агент должен сопоставить данные по школам, поликлиникам, паркам»
4. для каждого сценария зафиксируйте: какие данные нужны агенту и в каком виде (текст или число)

На диаграмме Context сама система (RAG-агент) изображается как единый прямоугольник. Вокруг него располагаются:

1. пользователи (акторы) — клиент, риэлтор
2. внешние системы — API Яндекс.Карт, API ЦИАН / резервный датасет, источник буклетов

> Диаграмма должна включать все внешние системы, зафиксированные в ТЗ на этапе 1.

#### 3.1.2. Классификация данных на два типа

Ключевое решение этапа — определить, какие данные попадут в векторную БД (семантический поиск), а какие — в структурированную витрину (точные расчёты и фильтрация):

- если данные — свободный текст, описание, рекламный материал, редко изменяются → **вектор** (буклеты, описания объектов)
- если данные — числа, категории, даты для точной фильтрации/сортировки/агрегации → **структурированное хранение** (цена, площадь, дата постройки, координаты)

Буклеты в формате TIFF — отсканированные изображения, прямое извлечение текста невозможно. Перед векторизацией такие файлы проходят OCR (например, Tesseract OCR, поддержка 100+ языков, форматы PNG/JPEG/TIFF). Пайплайн: TIFF-файл → OCR-модуль (Tesseract) → извлечённый текст → модуль векторизации → векторная БД. Для PDF, DOCX, PPTX текст извлекается напрямую (PyMuPDF, python-docx, python-pptx); OCR — только fallback для сканов без текстового слоя.

Отдельного внимания требуют текстовые данные по отзывам из соцмедиа — большой объём, неструктурированность, изменчивость. Обратите внимание на их актуализацию и целесообразность перевода в структурированный вид.

**Таблица «источник данных → тип хранения» (образец):**

| Источник данных | Тип хранения | Обоснование |
|---|---|---|
| Буклеты строительных компаний (pdf/docx/pptx/tiff) | Векторная БД | неструктурированный текст, нужен семантический поиск по описанию |
| Объявления о продаже (цена, площадь, тип) | Структурированная витрина | точные числовые фильтры и сортировка |
| Данные API Яндекс.Карт (расстояние до метро, объекты инфраструктуры) | Структурированная витрина | точные координаты и расстояния для расчётов |

> Таблицу нужно дополнить собственными строками по всем источникам из раздела 2.1.

### Этап 2. Проектирование технической архитектуры сбора данных (C4 Container). Требования к ИТ-инфраструктуре и защите данных

#### 3.2.2. Диаграмма уровня Container

Типовой набор контейнеров для гибридного хранилища RAG-агента:

1. модуль парсинга документов — извлекает текст из файлов разных форматов (pdf, docx, pptx, tiff)
2. модуль векторизации — преобразует текст в эмбеддинги
3. векторная БД — хранит эмбеддинги, обеспечивает семантический поиск
4. ETL-модуль сбора структурированных данных — собирает и очищает данные из внешних API/датасетов
5. реляционная витрина данных — хранит структурированные данные для точных запросов
6. модуль запроса внешних API — инкапсулирует обращения к внешним сервисам
7. оркестратор RAG-агента — объединяет результаты поиска из векторной БД и витрины, формирует запрос к LLM

**Алгоритм работы оркестратора (гибридный retrieval с предфильтрацией):**

1. из запроса клиента извлекаются структурированные фильтры (бюджет, район, тип, комнаты) — через function calling LLM или NER-модуль
2. по фильтрам выполняется SQL-запрос к витрине данных — получается ограниченное множество кандидатов (50–200 объектов)
3. внутри этого множества выполняется семантический поиск в векторной БД по свободной части запроса (векторный поиск с фильтром по ID из шага 2, а не по всей базе)
4. результаты семантического поиска и точные данные из витрины объединяются в единый контекст и передаются в LLM

> Такой порядок (сначала точная фильтрация, потом семантический поиск) обязателен, чтобы избежать ситуации, когда векторный поиск возвращает семантически похожие, но не подходящие по цене/району объекты.

**Задание:** постройте собственную диаграмму Container вашего варианта архитектуры (допускается объединение/разделение блоков, добавление кэширования, очереди сообщений — при обосновании).

*Рекомендуемые инструменты для C4: draw.io, Structurizr, PlantUML (C4-PlantUML), Miro.*
*Рекомендуемые инструменты для BPMN: Bizagi Modeler, Camunda Modeler.*

Сформулируйте и обоснуйте подходы к построению ИТ-инфраструктуры (собственные сервера, аренда виртуальных машин у провайдера и т.п.), а также требования к защите данных с учётом стоимости реализации, надёжности, нагрузки и законодательства по защите персональных данных.

> **Результат этапа 2:** диаграмма Container, описания алгоритмов в нотации BPMN с пояснительной запиской, описание требований к ИТ-инфраструктуре и защите данных. Обсудите архитектуру с преподавателем до перехода к этапу 3.

### Этап 3. Выбор технологического стека и проектирование ER-диаграммы

#### 3.3.1. Выбор векторной БД

Сравните минимум 3 варианта (Milvus, Qdrant, Chroma, Weaviate, FAISS) по группам критериев (не менее 12 критериев). Примеры критериев:

1. масштабируемость (распределённое хранение, миллионы векторов)
2. поддержка фильтрации по метаданным при поиске
3. интеграция с Python/LLM-фреймворками
4. производительность поиска (latency при заданном объёме векторов)

Каждый критерий должен быть измеримым (да/нет). Определите вес критерия (0–1, шаг 0,1). Составьте таблицу сравнения, просуммируйте оценки с учётом весов; решение с наибольшей суммой — оптимальное. Допускается выделение стоп-критериев (например, доступность решения в РФ).

#### 3.3.2. Выбор эмбеддинг-модели и СУБД для витрины

1. выберите модель векторизации (Sentence-BERT, bge-m3, text-embedding-3-small/large) — укажите поддержку русского языка и размерность вектора
2. выберите СУБД (PostgreSQL, MySQL — реляционная; ClickHouse — колоночная) с учётом объёма данных и характера запросов

Сравните минимум 2 модели и 3 СУБД по аналогии с выбором векторной БД.

#### 3.3.3. Проектирование ER-диаграммы витрины данных

Спроектируйте структуру витрины из минимум четырёх сущностей: «Объект недвижимости», «Район», «Инфраструктурный объект», «История цен». Для каждой определите:

1. первичный ключ (PK)
2. перечень атрибутов с типами данных
3. внешние ключи (FK)
4. тип связи (1:1, 1:N, N:M)

> **Результат этапа 3:** обоснование выбора стека (0,5–1 страница) и ER-диаграмма с текстовым описанием сущностей, атрибутов и связей.
