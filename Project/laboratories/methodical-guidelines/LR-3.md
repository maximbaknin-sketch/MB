## 1. Цель работы

Целью лабораторной работы является формирование у студента навыков проектирования и реализации ETL-процесса сбора и выверки данных из нескольких разнородных CRM-систем, включая ручную и автоматическую верификацию совпадений записей о клиентах (record matching), для последующего использования консолидированных данных в платформе анализа и прогнозирования продаж.

По итогам выполнения работы студент должен уметь:

1. Проектировать бизнес-процесс сбора и выверки данных в нотации BPMN (в редакторе Camunda Modeler).
2. Разрабатывать алгоритм маппинга записей о клиентах на основе ФИО и дополнительных атрибутов с учётом типовых ошибок написания.
3. Определять правило приведения ФИО к каноническому виду.
4. Программно реализовывать сбор и объединение данных из разных источников в единое хранилище.

## 2. Бизнес-контекст и входные данные

Заказчик — холдинг, включающий несколько бизнес-юнитов, каждый из которых ведёт учёт клиентов в собственной CRM-системе. Данные Заказчика прогоняются через Платформу анализа и прогнозирования, математический аппарат которой формирует на выходе таблицу с прогнозом, подтверждающим или опровергающим следующие гипотезы:

1. Насколько пересекаются клиенты одного бизнес-юнита с клиентами другого бизнес-юнита.
2. Насколько и как VIP-клиентов одного бизнес-юнита можно дополнительно развивать в другом бизнес-юните.
3. Насколько и как VIP-клиентов, переставших покупать в одном бизнес-юните, перевести в активные потребители в другом бизнес-юните.

*Например, по второй гипотезе на выходе может быть таблица, показывающая, какие покупатели с какой вероятностью (выше определённого порога) готовы покупать какой товар холдинга (из определённого в проекте перечня).*

В качестве исходных данных для платформы используются дата-сеты, имитирующие выгрузки реальных данных из двух CRM-систем холдинга. Выгрузка гарантирует, что платформа не будет вмешиваться в текущую работу самих CRM-систем. Состав и ретроспектива выгружаемых данных определяются, формируются и обосновываются студентом самостоятельно на основе экспертного понимания их ценности для анализа и могут меняться в процессе реализации проекта.

### 2.1. Outline-архитектура решения

Первоисточником «сырых» данных выступают две CRM-системы — мастер-системы управления данными о покупателях и покупках; покупатели в пилоте — преимущественно физические лица. Данные из эксплуатируемых систем имитируются наборами данных в реляционном виде (две базы данных), которые студент самостоятельно формирует на шаге 2.1. Из этих хранилищ данные могут загружаться в систему либо постоянно в режиме реального времени, либо через промежуточные хранилища или отдельными пакетами on-demand — в любом случае предполагается работа с большими объёмами данных.

## 3. Порядок выполнения работы

Работа выполняется поэтапно и построена как мини-проект интеграции данных. Каждый этап завершается обсуждением результата с преподавателем перед переходом к следующему шагу.

### Этап 1. Постановка задачи

На этом этапе необходимо формализовать задачу сбора данных из двух CRM-систем, включая проектирование полей целевой БД.

| Поле | Источник (CRM-1 / CRM-2) | Назначение в ХД |
|---|---|---|

#### 3.1.2. Зафиксируйте допущения и нефункциональные требования

Так как состав и ретроспектива выгружаемых данных могут меняться в ходе проекта, явно зафиксируйте в постановке задачи:

1. Глубину ретроспективы истории заказов (например, за последние 24 месяца).
2. Режим загрузки данных — потоковый (real-time) или пакетный (on-demand), и критерии выбора режима.
3. Ориентировочный объём данных на старте и ожидаемый рост.
4. Требование о неизменности работы исходных CRM-систем (выгрузка не должна создавать нагрузку на продуктивные системы).

### Этап 2. Проектирование бизнес-процесса сбора и выверки данных

На этом этапе необходимо спроектировать бизнес-процесс в редакторе Camunda Modeler, отражающий как автоматическую, так и ручную обработку данных при их сборе и выверке.

#### 3.2.1. Спроектируйте маппинг данных на основе ФИО

Основой сопоставления записей о клиентах из двух CRM-систем является ФИО, при необходимости дополняемое адресом и контактными данными. При проектировании обязательно учтите типовые варианты ошибок и различий в написании ФИО. Ниже приведён пример ошибок (в процессе выполнения работы расширьте его):

1. Разный порядок слов (Иванов Иван Петрович / Иван Петрович Иванов).
2. Неполнота написания (Иванов И.П. / Иванов Иван).
3. Смешение английских и русских букв (Ivanov / Иванов).
4. Двойные фамилии (Иванова-Петрова).
5. Разные варианты написания имени (Александр / Саша / Alex).

#### 3.2.2. Определите каноническое написание ФИО

Сформулируйте правило нормализации, приводящее любой вариант написания ФИО к единому каноническому виду. Пример правила: «Фамилия Имя Отчество», кириллица, полное написание (транслитерация латиницы в кириллицу по установленной таблице соответствий, восстановление полного имени по словарю сокращений при наличии однозначного соответствия).

Дополнительно потребуется словарь сокращений имён для восстановления полной формы (примеры пар для стартового словаря): «Саша» → «Александр»/«Александра», «Женя» → «Евгений»/«Евгения», «Дима» → «Дмитрий», «Настя» → «Анастасия», «Максим» — не сокращается. Такой словарь ведётся как отдельная справочная таблица и пополняется по мере обработки реальных данных; в случае неоднозначности (например, «Саша» может быть и мужским, и женским именем) запись автоматически направляется на ручную верификацию, а не приводится к канону автоматически.

Для транслитерации латиница–кириллица используйте официальную таблицу соответствий по ГОСТ 7.79-2000 (Система Б), фрагмент которой приведён ниже:

| б | b | л | l | ц | c, cz |
| в | v | м | m | ч | ch |
| г | g | н | n | ш | sh |

#### 3.2.3. Спроектируйте процесс в нотации BPMN

В Camunda Modeler постройте диаграмму бизнес-процесса, включающую следующие шаги:

1. Получение выгрузок из CRM-1 и CRM-2 (start-события).
2. Нормализация ФИО и адресов (автоматическая задача).
3. Сопоставление записей (matching) — расчёт меры схожести (confidence).
4. Шлюз (gateway) с несколькими исходами: совпадение найдено (confidence выше порога), совпадение неоднозначно, но возможно определение совпадения по дополнительным полям исходной базы, совпадение неоднозначно, но может быть определено с допустимой погрешностью (например, с учётом опечаток), совпадение не найдено.
5. Задача ручной верификации для неоднозначных случаев (user task).
6. Запись результата в хранилище данных (end-событие).

*Результат этапа 2: диаграмма бизнес-процесса в формате Camunda (BPMN XML или экспорт в PNG/PDF) с пояснительной запиской, описывающей правило канонизации ФИО и логику ветвления по confidence. Обсудите процесс с преподавателем до перехода к этапу 3.*

#### 3.3.2. Запрограммируйте сбор и объединение данных

Реализуйте алгоритм программно (рекомендуется Python/PySpark либо Scala/Apache Spark в соответствии с выбранным в разделе 2.1 инструментарием). Минимальный набор функций реализации:

5. Запись итоговых объединённых записей в целевое хранилище (файл/БД), включая маркер источника и статус объединения (авто/вручную).
